# Schema 3.0.0 acceptance testing

## Verified development workflow

On 2026-07-16, the `schema-3.0.0-testing` builder branch was tested against the `schema-3.0.0-development` validator branch using a real headless Chrome session and a comprehensive, self-contained fixture.

The test performed this complete workflow:

1. Open the browser metadata builder.
2. Import the complete schema 3.0.0 package fixture from `tests/fixtures/schema-3.0.0/pass`.
3. Confirm that the builder loaded one participant, one participant-characteristic row, one device, one device datasheet, one dataset, and the fully populated study contributor.
4. Confirm that the builder reported `Ready to export` with no missing required fields.
5. Download the metadata package ZIP generated by the browser.
6. Extract the ZIP.
7. Add the referenced `data/datasets/light.csv` file, because the builder deliberately does not bundle source data files.
8. Run the development validator against the exported `datapackage.json`.

The dataset light file contains two preamble rows, declares the true header at row 3, and contains typed and datetime-formatted values. The browser export preserved `dataset_file_header_row: 3`, and the validator skipped the preamble before validating the file.

Result:

```text
status: pass
schema_version: 3.0.0
error_count: 0
warning_count: 0
```

This verifies the browser importer, builder state mapping, metadata export, bundled schema paths, validator schema validation, cross-resource checks, and data-file content validation together.

The automated suite also covers valid and invalid alternatives that cannot coexist in one package, including participant and non-participant datasets, raw and processed files, fixed-interval and event-based timing, sensor and non-device modalities, column and collection datetime sources, all declared variable types, nullable core-resource fields, contributor requirements, participant constraints, and participant-characteristic foreign keys.

## Testing a real dataset package

1. Start the builder from the `schema-3.0.0-testing` branch.
2. Build or import the metadata and download the metadata package ZIP.
3. Extract the ZIP into the root of a copy of the dataset repository.
4. Place every source data file at the path declared by its `dataset_file_names` entry.
5. From the `schema-3.0.0-development` validator checkout, run:

```bash
VALIDATION_JSON=/tmp/validation.json \
VALIDATION_MANIFEST=/tmp/validated-files-manifest.json \
./.venv/bin/python gleam_validator.py /path/to/package/datapackage.json
```

Review `/tmp/validation.json` for errors and warnings. The manifest records the files and checksums included in that validation run.
